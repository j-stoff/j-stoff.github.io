Lesson 7 - Polymorphism

Polymorphism

The real benefit of Object orientation is the concept of polymorphism which works through inheritance. It allows the class which inherits from the parent class to have the functionality of the parent but allows for differentiation of the subclass. In better terms, a subclass isn't just adding a parent's code to it's own, instead it can change (morph) into more than one (poly) unique class.

This is the basic idea behind overriding though it wasn't revealed at the time. This brings us to defining code that may not be used much or even at all. In fact Java has a type of Object that can not be institated called an Abstract Object.

Abstract Object

What is an Abstract Object do?

As I said above, an object declared as abstract can not have it's constructor called. What is the point of having as class like that, if the code won't be used? Well, it is in fact used. It just isn't explicitly called anywhere.


public abstract class Person() {}

The code above is how to declare a class as abstract.

The benefit of an Abstract Object is that any inheriting classes will have the variables and/or methods already set up. It is often used generalized type that all classes have in common but doesn't make sense as a class itself.

For example: recall the Person > Employee > ITWorker example. Going above Person doesn't make sense for our program. We could call it 'Animal' or something else, but that isn't what we need in our program. If a Animal class was made it would likely be abstract as it doesn't need functionality in the program.

I still don't see the benefit. Isn't putting this logic in the sub-class easier?

If there were a direct one to one relationship, maybe. Instead, there could be several classes that inherit from this abstract class. Using the Animal > Person example above. What about adding a Dog class to this model? Or even a Cat class.

The point is that writing abstract classes allows inheritance to be taken at full value while not worrying about how the class should behave, as quite simply it shouldn't. It should just be a location which other classes pull code from.

Abstract methods

Abstract classes do not need to contain abstract methods but often do. An abstract method has no formal definition except the access modifier, return type, and parameters. No method body means that calling the method would do nothing which seems fairly pointless. The idea is that all abstract methods must be defined by inheriting classes.

An example might be an abstract data structure in Java called List. The List class defines some basic functionality you might expect in lists: an add() method, a remove() method, a size() method to get the current side of the list. Subclasses of List inlcude ArrayList and LinkedList, so both of these classes must define add(), remove(), and size() methods. The implementation for add() and remove() is very different for an array based list versus a linked list. Side note, List is an abstract type that can be used as a variable type but not as a class declaration.

Interfaces

Interfaces are abstract code that is meant to be specific for the class that is implementing it. Speaking of implements is the keyword that allows inheritance of an interface to a class. A class can implement many interfaces and an interface can be implemented by many classes. This is one of the real benefits to Object orientation as it allows some baseline code to be written in an interface then any other developer can write their own classes that implement the interface.

Because an interface is inherently abstract, that means an interface can never be instantiated. It also means that all methods are treated as abstract and all varaibles are treated as final (they must be assigned a value that can not change).

Look at this example of an interface.


interface Collection {
    int MAXIMUM_SIZE = 1000;

    void add();
    Object remove();
}

Notice how there are no access modifiers meaning that the default mode is being used. Generally public is declared as it allows unresticted access to these fields and methods. Interfaces can use any of the modifiers, but private would not make much sense as they woud be unusable. The exception is that in Java 9, interfaces can actually fully declare their own methods. Not just abstract methods, but that is still to come.
