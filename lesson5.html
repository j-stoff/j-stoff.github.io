Lesson Five

Object orientation: A big deal

I swear its getting hard coming up with good titles.

Object orientation is the heart of programming with most modern languages. It isn't required by some, but is helpful nonetheless.

Why is it helpful? Code reusability. That's it in a nutshell. One class definition can detail everything several functions could do but keeps the code nice and neat.

Recall the definition I gave for Objects. Containers for code, and that is exactly what they are. Useful things to store (varaibles), and useful functions (methods).

Why do I care? Because of Inheritance and Polymorphism

We will start with Inheritance

So if the heart of Object orientation is code re-usability, can we use classes we've already written. Yes.


                public class Person {
                    String name;
                    int age;

                    public void sayHello() {
                        System.out.println("Hello");
                    }
                }

The class above is fairly simple with 2 instance variables and only one method. Not much, but there are classes just like this in huge applications.

The reasons may differ, sometimes it could be a base class that a series of other classes are based on. Or it could just be one or two useful methods you call over and over but dont fit in any particular class thus far.

Re-using classes: extends keyword

Java uses the keyword extends to declare a super class with which to derive code from.


                public class Employee extends Person {
                    int id;

                    public void startShift() {
                        // Code
                    }
                }

The implications are as you might expect. An Employee reference type can use all the employee fields (the id) and methods (startShift). But it can also use those from the Person class. The reverse is not true. A Person class cannot call startShift() as the Person class knows nothing about Employee. It only the Employee that knows about the Person reference.

Also a note: There can only be one super class to inherit from. There is a way to use code from multiple sources - it's polymorphism we will get to later.

Simple test

Often it is easy enough to say I want this class to have the same methods/code as this other class. The issue becomes, should it?

Logical design in class structure is more art than actual design. It takes time and practice for what classes should be used and how the hierarchy should be structured.

That said, there is a logical test to do that works quite well. The IS-A test, which is very simple itself. If the inheriting class name (sub-class) 'Is A' superclass name. If the statement makes sense, it passes the test.

Example: An Employee is a Person. A Person isn't always an Employee.

There is also another simple test to determine if the relationship should be through a variable (reference to the object). The HAS-A test is similar to IS-A, but if the class in question 'has a' other class name, it fulfills the has-a test.

Example: An Employee class has a Job Class. The Job class doesn't extend from the Employee class, as a Job IS-AN Employee wouldn't make sense.

But we work with classes, what's all the talk about Objects.

Well, recall that I said the Object in Java is the base level from which everything is bult? That means every class EXTENDS from object though it doesn't need to declare it. That means any Object methods are usable by any class (mainly the .equals() and .toString())

How large can be hierarchy of super and sub classes be?

As long as you want. Though there is a better way, polymorphism, to reuse some common code that doesn't require extremely long chains of inheritance.

In truth, inheritance is pretty cool, but it's become the less useful feature of Object Orientation. Designing through Interfaces, or generically in any language as polymorphism, is the real benefit.

Side note: It is possible to compile multiple .java files by using


                javac *.java

Which will compile all .java files in the current directory and along the java classpath.
